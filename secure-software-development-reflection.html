Final Reflection

My Experience

This module focused on the principles of secure software development, emphasising the importance of integrating security throughout the software lifecycle rather than treating it as a separate or final-stage concern. Across Units 1–12, the module introduced a progression from theoretical security concepts to their practical implementation within a working system.

In the early stages of the module, I explored key concepts such as the OWASP Top 10 and STRIDE threat modelling. These frameworks provided a structured approach to identifying vulnerabilities, including injection attacks, brute-force authentication attempts, and denial-of-service risks (OWASP, 2021). Initially, these models felt quite abstract, and I found it difficult to directly connect them to real software systems.

This understanding developed further during the team-based design document, where we proposed a secure online retailer system. The design applied the Model–View–Controller (MVC) pattern alongside a Strategy Pattern to modularise security behaviour (Lott and Phillips, 2021). A central feature was the secure_mode toggle, which allowed the system to switch between secure and insecure configurations. This approach enabled direct comparison of how different security controls affect system behaviour. The design also included authentication controls, role-based access control, input validation, and simulated external API integration.

In the later stages of the module, I developed an individual implementation of the system using Python and a command-line interface. The application included user registration, login, product management, and order processing, supported by security mechanisms such as input validation, salted password hashing using PBKDF2, account lockout, session management, and audit logging. The system retained the secure_mode functionality, allowing the behaviour of secure and insecure configurations to be demonstrated in practice, reflecting security-by-design principles (Olmsted, 2024).

Development and Learning Process

This module significantly changed my approach to software development. Previously, my focus was primarily on ensuring that systems worked correctly and were structured effectively using object-oriented principles. However, this module introduced the need to consider how systems can fail, be exploited, or misused, and how those risks can be mitigated through design.

One of the most important learning outcomes was understanding how theoretical security frameworks translate into practical implementation. For example, identifying brute-force attacks during the design phase directly informed the implementation of account lockout mechanisms in my application. Similarly, recognising injection risks led to the use of strict input validation rules. This demonstrated that security is not just conceptual but directly influences how code is written and structured (OWASP, 2021).

The transition from the design document to the individual implementation was particularly valuable. The original design proposed a more comprehensive system, including full API integration and broader mitigation strategies such as request throttling. However, during implementation, I had to prioritise key features that could realistically be developed within the given timeframe. This aligns with the idea that secure systems must balance security, usability, and performance (Olmsted, 2024). As a result, I focused on core security mechanisms such as authentication, validation, and access control, while simplifying other aspects, such as payment processing through a simulated gateway.

From a technical perspective, implementing authentication and password security was especially insightful. Using PBKDF2 with a salt demonstrated how secure password storage differs from basic hashing techniques. Additionally, implementing account lockout and session expiry introduced real-world considerations such as managing state and balancing usability with security. These features were more complex than expected and required careful testing to ensure they behaved correctly.

There were also challenges during development. Debugging authentication logic and handling edge cases, such as repeated failed login attempts, was sometimes frustrating. Small errors often caused unexpected behaviour, which required time to diagnose and resolve. However, these challenges improved my problem-solving skills and encouraged a more systematic approach to debugging and testing.

Engagement with discussion forums also contributed to my learning. Reading other students’ perspectives on topics such as encryption tools and system vulnerabilities helped broaden my understanding of security beyond technical implementation. It became clear that user behaviour and usability play a significant role in the effectiveness of security controls, which aligns with reflective learning principles (The University of Edinburgh, 2024).

Compared to my previous modules, this experience required a deeper level of critical thinking. It was not enough to build a functional system; I needed to justify design decisions in terms of security risks and real-world applicability. This shift has been an important step in my development as a software engineer.

Going Forward

The knowledge gained from this module will strongly influence my future approach to software development. I now understand that security must be considered from the beginning of the design process rather than added later. Applying security-by-design principles will be essential in developing systems that are both functional and resilient (Olmsted, 2024).

From a technical perspective, I plan to build on this foundation by exploring more advanced areas such as secure API development, database security, and multi-factor authentication. While this project used a JSON-based repository for simplicity, I recognise that more robust data storage solutions would be required in real-world applications.

I also aim to improve my testing practices. Although I used pytest and linting tools effectively, I would like to expand into more comprehensive testing strategies, including integration testing and automated security testing. This reflects industry practices where testing is essential for validating both functionality and security.

Reflecting on the development process, I have learned the importance of balancing ambition with practicality. While the design phase encouraged comprehensive thinking, the implementation phase required prioritisation. In future projects, I would aim to define clearer boundaries between essential and optional features to ensure efficient delivery.

In terms of collaboration, the team design phase and discussion forums demonstrated the value of shared perspectives. Engaging with others provided new insights and helped challenge my assumptions. I would like to participate more actively in collaborative development environments in the future.

Overall, this module has significantly enhanced both my technical skills and my understanding of secure software development. It has shifted my focus from simply building functional systems to designing systems that are secure, resilient, and aligned with real-world requirements. I feel more confident in my ability to apply these principles in future academic and professional work.
